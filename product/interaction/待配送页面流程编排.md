# 待配送页面流程编排（基线）

## 1. 目标与范围

围绕“待配送”主链，先固定页面流程与职责边界，避免把交易、催收、库存、改单继续挤在单页。

本编排仅覆盖配送员高频执行链路：
- 待配送列表
- 完单提交
- 安检闭环
- 24h 改单回流

## 2. 页面流转总图

```text
工作台（今日待配送卡）
  ├─ 查看全部待配送 -> 订单列表（筛选 pending_delivery）
  │    ├─ 点击“送达收款” -> 待配送完单页（delivery-complete）
  │    └─ 点击订单行 -> 订单详情页（order-detail）
  │
  └─ 直接开始配送 -> 待配送完单页（delivery-complete?view=pending）

待配送完单页（交易执行）
  ├─ 完单成功 -> 安检提交/重试（同页闭环）
  ├─ 去订单详情改单 -> 订单详情页（24h 改单）
  ├─ 去债务管理 -> 债务管理页（催收/还款）
  └─ 返回 -> 工作台（或来源页）

订单详情页（交易后管理）
  ├─ 24h 改单（基础信息/金额）
  ├─ 退货/换货
  └─ 返回 -> 来源列表（保持滚动位置）
```

## 3. 页面职责边界

### 3.1 待配送完单页（delivery-complete）

只做：
- 选择待配送订单
- 完单输入与提交（实收、收款方式、回收空瓶、欠瓶）
- 5 秒撤销
- 安检提交与重试（按策略）

不做：
- 催收状态编辑
- 库存全量快照浏览
- 24h 改单完整操作

### 3.2 订单详情页（order-detail）

只做：
- 订单交易后管理
- 24h 改单（统一入口）
- 退货/换货

不做：
- 批量待配送执行（这是完单页职责）

### 3.3 债务管理页（debt-overview）

只做：
- 催收状态维护
- 催款动作记录
- 还款登记

不做：
- 完单交易动作

### 3.4 库存管理页（inventory）

只做：
- 库存快照、盘点、补录

不做：
- 完单交易动作

## 4. 返回与导航一致性规则

- 有来源页时：优先 `goBackOr(来源页)`。
- 无来源页时：
  - 完单页返回工作台
  - 订单详情返回订单列表
  - 债务管理返回“我的”
- 不在业务页重复放“返回首页”类捷径；依赖底部导航与统一 back 逻辑。

## 5. 状态约束（待配送主链）

- `pending_delivery -> completed` 是唯一完单状态跃迁。
- 若 `safetyCheckRequired=true`：
  - 上一笔完单安检未完成时，阻断下一笔完单。
- 完单后：
  - 默认提示 5 秒撤销；
  - 24h 改单只在订单详情页执行。

## 6. 数据口径（待配送链路最小集）

- 列表/详情/待配送同口径：`OrderContract`（camelCase）
- 完单提交：
  - `receivedAmount`
  - `paymentMethod`
  - `recycledEmptyCount`
  - `owedEmptyCount`
- 安检提交：
  - `checkItems`
  - `photoUrls`
  - `hasAbnormal`
  - `hazardNote`

## 7. 基于本流程的下一步任务包（待配送方向）

1. `待配送入口统一`
- 统一工作台、订单列表到完单页的入参与返回回路。

2. `完单页再瘦身`
- 仅保留执行链与安检闭环，进一步清理非执行信息噪音。

3. `订单详情改单体验`
- 24h 改单交互补齐：字段约束、修改记录、失败提示一致化。

4. `待配送全链路故事回归`
- 入口 -> 完单 -> 安检 -> 改单回流 -> 催收分流，做端到端回归包。
