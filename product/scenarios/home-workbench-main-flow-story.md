# 首页工作台主链路用户故事

## 1. 用户与场景

- 用户角色：液化气配送员
- 使用设备：移动端浏览器，390x844 视口优先
- 使用环境：出车前、配送间隙、收工前快速查看经营状态
- 触发时机：打开 App 后默认进入首页；用户想知道今天收入、下一单、是否要继续开单

## 2. 用户目标

- 3 秒内知道今天赚了多少、还有没有待送任务
- 不经过管理页，直接进入快速开单或待配送流程
- 结束配送后能顺手进入收入明细或日结，而不是再找财务入口

## 3. 主路径

1. 用户进入首页，首屏先看到今日收入摘要。
2. 首页紧接着给出两个高频动作：快速开单、待配送。
3. 用户向下看到下一配送卡片，决定直接开始配送或查看全部待配送。
4. 用户若要核对经营结果，从收入卡直接进入收入明细或日结，不需要先进入“我的”查找。

## 4. 风险与异常

- 首页如果混入登录、同步、平台入口，会破坏“开工即干活”的心智。
- 首页如果只展示收入，不提供去财务轻入口，用户会觉得卡片只是装饰。
- 用提示文案去弥补结构问题，会导致页面看起来像说明书，而不是工作台。
- 离线与同步异常应作为状态反馈出现，不应长期占据首页主卡位。

## 5. 页面职责建议

- 入口归属：底部 Tab 的“首页”
- 主页面职责：
  - 收入摘要
  - 高频动作
  - 下一配送与待配送入口
- 不应承载的职责：
  - 登录与账号切换主入口
  - 同步队列常驻入口
  - 平台运营跳转
  - 低频功能转移说明文案

## 6. 验收故事

- 当用户打开首页时，应先看到收入、主操作、下一配送，而不是设置或账号操作。
- 当用户点击收入卡时，应能直接进入收入明细或日结相关页面。
- 当用户离线或同步异常时，应看到状态反馈，但不应因此改变首页主结构。
